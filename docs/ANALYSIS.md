## 1. 기획 의도 — README가 설계한 3개의 루프

README를 한 문장으로 압축하면: **"검증 가능한 가장 좁은 시장(요양병원 야간 간호사)에서, 매칭이 아니라 '정산 인프라'로 돈을 버는 사업"**.

| 루프 | 기획 | 왜 중요한가 |
|---|---|---|
| **매칭 루프** | 위치 기반 실시간 매칭 + 푸시 | 공급(간호사)을 모으는 미끼 |
| **신뢰 루프** | QR 체크인 + 위치 검증 + 자동 정산 | 직거래 이탈을 막는 해자 — "우리 통해야 근태·정산이 깔끔하다" |
| **돈 루프** | 선불 크레딧(Float) + 12% 수수료 + 즉시정산(EWA) | 진짜 수익원. 매칭 수수료는 입구, EWA가 캐시카우 |

설계 자체는 탄탄하다. ShiftKey($2B) 검증 모델이고, "매칭 앱"이 아니라 "정산 시스템에 매칭이 붙은 것"으로 설계한 것이 핵심 차별점.

## 2. 구현 현황 — 루프별 대조 (2026-07-10)

### 매칭 루프: 기획 대비 100% + α ✅
- 기획은 선호 지역 기반 매칭 → 구현은 GPS 현재위치까지 이중 기준 (`get_nearby_open_shifts_v2`). 기획서보다 앞서감
- 푸시 알림(Web Push), 필터(날짜/시간대/시급/부서), 원터치 지원 완성

### 신뢰 루프: 핵심 완성, 외곽 대기 🟡
- QR 체크인 + **GPS 지오펜스 500m** + 자동 임금계산(연장/야간 +50%, 휴게 차감, 2026 포괄임금 금지 대응) ✅ — 영업 멘트의 심장
- 회전 QR·HMAC·Nonce는 스키마만 준비 — 스크린샷 부정은 아직 뚫림
- 면허 검수 수동 (Phase 1 기획과 일치 — 문제 아님)
- 본인인증(PASS)·계좌 실인증(1원)은 사업자등록 대기 — 코드로 할 일은 완료

### 돈 루프: 기획과 구현이 서로 다름 ⚠️ → 3장

## 3. ⚠️ 핵심 발견 — 수익 모델이 두 개 존재한다

README 기획과 현재 코드가 **서로 다른 수익 구조**를 갖고 있다:

| | README 기획 (섹션 9·10) | 실제 구현 (2026-07-10) |
|---|---|---|
| 크레딧 | 100만 결제 → **107만** 지급 (+보너스 = 미끼) | 100만 결제 → **94만** 지급 (할인 마진) |
| 수익원 | 시프트당 **12% 매칭 수수료** + Float 이자 | **크레딧 스프레드 2~3.6%** 뿐 |
| 정산 시 | 임금 + 12% 수수료를 크레딧에서 차감 (기획 의도) | **임금(gross)만 차감** — 수수료 차감 코드 없음 |

`apps/admin-web/app/checkin/actions.ts`의 `credit_ledger` 차감은 `-gross`만 기록하며, **기획의 핵심 수익인 12% 수수료가 어디에서도 걷히지 않는다.** 2026-07-09의 크레딧 티어 흑자 개편은 "Toss 3.4% 수수료로 손해 안 보기"였지 12% take rate 회복이 아니었다.

**숫자 비교**
- 기획 유닛 이코노믹스: 시프트 1건당 마진 1.8~2.5만원 (take rate 7.4%)
- 현재 구현: 100만원 크레딧 소진 전체에서 2.6만원 — **기획 대비 수익 약 1/10**

**부수 불일치**
- README 섹션 9 티어 표(107만/+7% 등)는 구버전 — `lib/billing.ts`와 불일치
- 환불 약관 "원금 기준 환불"이 이제 모호 — 현재 구조는 원금(결제액) > 지급 크레딧

### 선택지

**A. 기획대로 (권장)** — 크레딧 100만→100만 지급, 체크아웃 정산 시 `gross × 1.12` 차감 (임금 + 수수료).
- 장점: 12% take rate 회복, 티어 보너스 기획대로 부활 가능, 유닛 이코노믹스 성립
- 단점: 수수료가 병원 눈에 명시됨 (단, 업계 표준이라 설득 가능)

**B. 현행 유지** — 수수료 없는 대신 크레딧 할인 (마진 2~3.6%).
- 장점: "수수료 0원" 영업 후킹
- 단점: CS·산재·분쟁 운영비(기획 추산 건당 5천원)를 못 덮음 — 스케일할수록 손해

→ 코드로는 반나절 거리지만 **사업 결정 사항**. 결정 후 정산 로직 + README 섹션 9 + 환불 약관을 한 번에 동기화할 것.

## 4. UI/UX 퍼널 분석

### 워커 퍼널
`카카오 → 약관 → 직군 → 지역 → 면허(스킵 가능) → 본인정보 → 계좌 → 심사`

**잘한 것**
- 단계당 질문 1개 원칙, 면허 스킵 허용
- 가짜 인증 화면(신분증/1원 OTP) 제거로 신뢰 리스크 해소 (2026-07-10)

**마찰 1 — 계좌를 가입 시점에 요구**
아직 일자리도 못 봤는데 계좌부터 요구. 첫 매칭 확정 후 "정산받으려면 계좌 등록"으로 미루면 온보딩 전환율 상승 + 등록 동기 강화 (받을 돈이 실재하는 시점).

**마찰 2 — 심사 대기 블랙홀**
승인까지 워커가 할 게 없음. 대기 중에도 시프트 열람은 허용(지원만 잠금)하고, 승인 시 푸시로 재유입시켜야 이탈 방지.

### 관리자 퍼널
`카카오 로그인 → 병원 연결 → 시프트 등록(30초) → 지원자 수락 → QR 스캔 → 자동 정산`

**잘한 것**
- 40대+ 타깃 큰글씨 토글, 시프트 등록 최소 입력
- 대시보드 "충전 필요" 선제 경고 + 추천 티어

**구조적 리스크 — 야간에 누가 스캔하나**
1순위 타깃이 *야간* 간호사인데, 체크인은 *관리자 기기*가 워커 QR을 스캔하는 구조. 야간엔 원무과가 비어 있는 요양병원이 많다.
→ **역방향 체크인 추가 권장**: 병원 벽 고정 QR 게시 → 워커 폰이 스캔 → 워커 GPS로 지오펜스 검증. 서버 지오펜스 로직(WKB 파싱 + 하버사인)은 그대로 재사용 가능.

## 5. 비즈니스 관점 평가

- **해자는 정산에 있다** — 매칭은 베끼기 쉽지만 "체크아웃 순간 가산수당까지 계산 완료 + 2026 포괄임금 금지 대응"은 요양병원 원무과의 실제 고통을 지운다. 영업 데모의 클라이맥스는 이 장면(체크아웃 → 급여 자동 산출)이어야 한다
- **Lock-in 체인 구현됨**: 선불 크레딧(잔액) → 근태 기록 축적 → 급여 명세 이력. README "멈춰야 할 신호" 중 직거래 이탈 50%+에 대한 방어가 실제로 코드에 있다
- **규제 체크**: 유료 직업소개는 **직업안정법상 등록 대상** (유료직업소개사업, 관할 구청). 사업자등록과 함께 토스 전자결제보다 먼저 처리하는 것이 순서상 안전
- **계기판 부재**: README의 "멈춰야 할 신호"(3개월 베타 30건 미만 등)를 잴 지표가 없음 — 주간 시프트 수 / 매칭률 / 직거래 의심(반복 매칭 후 이탈) 3개만 어드민에 추가하면 됨
